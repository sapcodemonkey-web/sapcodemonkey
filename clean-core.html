Explainer · Clean Core & ABAP Cloud

Clean core isn't no customisation. It's customisation that survives the upgrade.

The phrase gets used as if it means "stop building things." It doesn't. It means building them where the next release can't break them — and SAP now has a four-level scale for telling the difference. Here's the whole model, including the part most explainers haven't caught up with.

The problem it solves

Twenty years of "just add an exit"

Every modification, implicit enhancement and direct table write was individually reasonable and collectively fatal. Upgrades became archaeology projects. Systems got frozen because nobody could predict what a patch would break. The core stopped being standard, so standard innovation stopped arriving.

What changes

Extensions that hold a contract

Build against interfaces SAP has formally committed to keeping stable, and your extension keeps working across releases. Build against internal objects, and you own the risk every time SAP ships. The difference isn't whether you customise — it's what you customise against.

Wider than code

Six dimensions, not one

Developers hear "clean core" and think custom code. That's one dimension of six — and often not the one holding a project back.

Software stackStay on supported releases, no modifications to the standard.
ExtensibilityHow and where you build custom logic. The A–D levels below live here.
IntegrationsReleased APIs and governed interfaces rather than point-to-point improvisation.
ProcessesStandard where standard works. Divergence is a cost, not a default.
DataQuality and volume under control. Archive what you don't need.
OperationsPredictable, monitored, upgrade-ready as routine rather than as a project.

The rule that catches people out

An extension is graded by the lowest-ranked thing inside it. One direct table write in an otherwise immaculate ABAP Cloud application drags the whole thing to Level D. There is no averaging, and no partial credit — which is exactly why "mostly clean" isn't a status anyone should be reporting.

If you learned this before 2026

The model you know was replaced

Plenty of training material, blog posts and consultants still teach three tiers. It isn't wrong exactly — it's superseded.

Until August 2025 The three-tier model

Tier 1 was ABAP Cloud and BTP. Tier 2 was wrappers around SAP objects not yet released. Tier 3 was classic ABAP. Tier 1 and some of Tier 2 counted as clean; Tier 3 was simply "not clean."

→
Current Clean core levels A–D

A four-point maturity scale graded on upgrade safety rather than a binary verdict. Classic ABAP is no longer uniformly condemned, more released APIs mean fewer wrappers, and every extension gets a specific grade you can track and improve.

Why the change matters practically: the old model told you your Tier 3 estate was "not clean" and left you there. The level scale tells you whether a given object is a documented-API dependency you can live with (B), an internal-object dependency needing a roadmap (C), or a modification that needs a retirement date (D). Same estate, far more actionable — and a remediation backlog you can actually sequence.

Practical

What to do about it

Not a transformation programme. Five things that move the needle.

01

Grade what you already have

Run ATC with the clean core and ABAP Cloud readiness checks against your custom code. You'll get an inventory graded by level. Most teams discover their estate is less uniformly bad than feared and more concentrated in a handful of objects than expected.

02

Stop the bleeding before you fix the backlog

New development goes to Level A. Set ATC to block Level D patterns outright in new code. Remediation is slow; preventing additions is immediate, and it stops the backlog growing while you work on it.

03

Triage by risk, not by count

Level D objects touching processes you run monthly matter far more than Level C objects in a report nobody opens. Retire the dead code first — that's usually a surprising share of the estate and the cheapest win available.

04

Wrap what you can't yet replace

Where a released API doesn't exist, isolate the dependency behind your own interface. Your code calls the wrapper; the wrapper calls the internal object. When SAP releases a proper API, you change one thing rather than fifty. Retire the wrapper at that point.

05

Decide on-stack versus side-by-side deliberately

ABAP Cloud on-stack when the logic genuinely needs transactional proximity to the data. BTP side-by-side when it doesn't. Both reach Level A — the choice is architectural, not compliance-driven, and treating it as compliance leads to things running in the wrong place.

Before you quote any of this: clean core guidance is revised regularly — the A–D model itself replaced the three-tier framing in August 2025, and the certification and tooling around it moved again during 2026. Check the current ABAP extensibility guide and SAP's clean core documentation for your release before making commitments in a project plan.

The migration wave behind all this
